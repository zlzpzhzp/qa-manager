## C2. `simpleHash()` — HMAC이 아닌 32bit 해시로 토큰 서명
- 파일: `src/lib/supabase.ts:19-31`
- 현상: `signToken(payload) = simpleHash(payload + AUTH_SECRET + payload)` 구조. `simpleHash`는 djb2 계열 shift-and-add로 `hash & hash`로 32bit 정수 변환 후 `Math.abs`로 31bit 양수화. 최종 출력 공간 ~2^31. secret을 모르더라도 31bit 서명을 브루트포스로 충돌시키는 것이 이론적으로 가능(~21억 시도). 또한 djb2는 prefix-extension 공격에 취약한 계열. 테스트(`auth.test.ts`)는 단순 변조만 확인하고 충돌 위조는 검증하지 않음.
- 원인추정: incident `04-02~04.암내.vercel_root_cause`에서 "Node crypto→빌드실패 → Web Crypto/simpleHash"로 결정된 흔적. Edge runtime에서는 `crypto.subtle.sign('HMAC', ...)` 즉 WebCrypto API가 정상 동작하므로 simpleHash로 후퇴할 이유 없음. incident `04-14.워라.cookie_forgery`와 정확히 같은 부류("인증있다≠인증작동한다").
- 해결옵션:
  1. WebCrypto HMAC-SHA256으로 교체 — `const key = await crypto.subtle.importKey('raw', encoder.encode(secret), { name: 'HMAC', hash: 'SHA-256' }, false, ['sign', 'verify'])`. edge/serverless 양쪽 호환.
  2. `jose` 라이브러리로 JWT HS256 — 만료/재발급/커스텀 클레임 표준화. 다른 앱과 통일 가능.
  3. Supabase Auth 활용 — `supabase.auth.signInWithPassword({ email: 'teacher@dm', password: PIN })` + 세션 쿠키. 장기적으로 가장 건전.

## C4. RLS 정책이 공개 anon key에 전체 DELETE 허용
- 파일: `supabase-schema.sql:22-24`
- 현상: 정책 `for delete using (true)`로 모든 사용자에게 삭제 허용. 공개된 `NEXT_PUBLIC_SUPABASE_ANON_KEY`로 다음과 같이 API 우회 가능:
  ```
  curl -X DELETE 'https://pasycnvfdotcdzzysqbz.supabase.co/rest/v1/qa_submissions?id=gte.0' \
    -H "apikey: sb_publishable_..." -H "Authorization: Bearer sb_publishable_..."
  ```
  → 전체 행 삭제. 주석 "서버에서 auth 검증"은 잘못된 전제 — Supabase REST는 Next.js를 거치지 않음. SELECT/INSERT도 동일 공격 가능. `__tests__/submissions.test.ts`가 실제로 이 경로로 prod DB를 건드리고 있어 증거가 됨.
- 원인추정: 초기 MVP에서 "학생이 insert만 할 수 있으면 됨" 로직을 서버측 API로 밀어넣으려 했는데, RLS는 "누구나"로 열어놔서 앱 단계가 아닌 DB 단계 통제가 무력화됨.
- 해결옵션:
  1. RLS를 서비스 롤 전용으로 좁히기: `create policy "service only delete" on qa_submissions for delete to service_role using (true);` → anon은 삭제 불가. 서버 API에서 `createClient(url, SERVICE_ROLE_KEY)` 사용. anon 클라이언트는 INSERT만.
  2. SELECT도 RLS로 제한: 선생님용 SELECT는 서비스 롤로만, 학생 페이지는 SELECT 불필요. 최소권한 원칙 준수.
  3. INSERT에 `with check` 강화: `class_name in ('중1S', ..., '고3K')` 및 `char_length(content) < 2000` 추가.

## H1. 인쇄 iframe에서 DOM XSS
- 파일: `src/app/teacher/page.tsx:177-263` (`handlePrint`)
- 현상: `doc.write(\`...\${originalLines.join('\\n')}...\${analysis}...\${verification}...\${correctionsHtml}...\`)` 구조. `originalLines`는 `s.student_name`, `s.content`를 그대로 포함, `classificationHtml`도 `q.textbook`, `q.label`, `q.students` 및 name을 템플릿 리터럴로 주입. 학생이 `<img src=x onerror=alert(1)>`을 content에 제출하면 선생님이 인쇄 버튼 클릭 시 iframe에서 실행됨. iframe은 same-origin이므로 쿠키(teacher_auth) 탈취 → 세션 위조(C2와 결합 시 더 치명).
- 원인추정: 직렬화가 `<pre>` 태그 안이라는 오해. 실제로는 `<pre>` 안에서도 `<script>` 태그는 파싱됨(HTML 파서 관점). 더욱이 `classificationHtml`는 `<div>` 블록으로 들어가므로 일반 태그 모두 파싱.
- 해결옵션:
  1. 이스케이프 헬퍼 도입: `const esc = (s: string) => s.replace(/[&<>"']/g, c => ({'&':'&amp;','<':'&lt;','>':'&gt;','"':'&quot;',"'":'&#39;'}[c]!))`. 모든 동적 문자열에 적용.
  2. `doc.write` 폐기 → `doc.body.appendChild(document.createElement(...))`로 DOM API 직접 사용. textContent는 자동 이스케이프.
  3. 서버사이드에서 PDF 생성 (puppeteer, react-pdf) → 클라이언트 iframe 제거로 공격면 자체 삭제.

## H2. "전체 삭제" 버튼이 실제로 400 에러 반환
- 파일: `src/app/teacher/page.tsx:115-122`, `src/app/api/submissions/route.ts:56-71`
- 현상: `handleDeleteAll`은 `fetch('/api/submissions', { method: 'DELETE' })` — 쿼리스트링 없음. 백엔드는 `const id = url.searchParams.get('id'); if (!id) return 400 "id 파라미터 필수"`. 즉 전체 삭제 UI는 동작하지 않으며, 선생님은 "삭제 성공" 시각적 피드백 없이 목록이 그대로 남음을 보고 실제로는 `fetchSubmissions()`가 다시 호출되어 목록 재표시되는 현상을 삭제 후로 착각할 수 있음.
- 원인추정: 초기에 단일 DELETE 엔드포인트를 재사용하려다가 id 없을 때의 경로를 구현하지 않음.
- 해결옵션:
  1. API 분기 추가: `id`가 없으면 `supabase.from('qa_submissions').delete().neq('id', 0)` 또는 날짜 범위 기반 대량 삭제(더 안전). 200 반환.
  2. 별도 엔드포인트 `/api/submissions/bulk` 또는 `/api/submissions/purge` — 의도를 명확히 하고 대량 삭제에만 별도 권한 검사.
  3. UI에서 현재 필터된 submissions 목록을 순회해 개별 DELETE — 네트워크 비용 크나 기존 API 수정 없이 동작.

- 현상: `process.env.SUPABASE_SERVICE_ROLE_KEY || 'fallback-secret'`. `SUPABASE_SERVICE_ROLE_KEY`는 서버 전용이며 client bundle에 `undefined`로 대체됨. 즉 client가 이 모듈을 runtime import하면 secret은 `'fallback-secret'` — 공개된 고정값. C2와 결합 시 클라이언트에서 유효 토큰 생성 가능.
- 원인추정: 빌드 시 env 미세팅으로 인한 crash를 피하려는 방어.
- 해결옵션:
  1. 서버 전용 모듈로 분리: `src/lib/auth.server.ts`로 이동 + `'server-only'` 패키지 import로 client 번들 혼입 차단.
  2. fallback 제거 + 서버 부팅 시 assert.
  3. 별도 env `AUTH_SECRET` 전용 변수 도입 (서비스 롤 키 재사용 중단).

## M6. `useEffect` 재조회 + "조회" 버튼 중복
- 파일: `src/app/teacher/page.tsx:73-92, 352-357`
- 현상: `fetchSubmissions`는 모든 필터 값에 의존하는 useCallback. `useEffect`는 fetchSubmissions 변경마다 재실행 → 필터 한 글자 타이핑마다 네트워크 호출. 동시에 "조회" 버튼이 존재. 수동 조회와 자동 조회 의도가 충돌.
- 원인추정: 점진적 UX 변경.
- 해결옵션:
  1. 자동 조회 제거 → "조회" 버튼 클릭 시에만 실행. useEffect는 초기 1회만.
  2. 입력에 debounce 300ms 적용 (lodash 없이 useDeferredValue 또는 setTimeout).
  3. React Query 도입으로 캐싱/중복요청 방지.
